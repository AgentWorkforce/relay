---
title: 'Spawning an agent'
description: 'Launch Claude, Codex, Gemini, or OpenCode agents from TypeScript or Python, then wait, message, and manage them cleanly.'
---

Spawning starts a CLI-backed worker on the relay and returns an `Agent` handle you can wait on, message, observe, and release later.

Use it when your application needs to create workers on demand instead of asking the user to launch Claude Code, Codex, Gemini, or OpenCode manually.

## Spawn APIs

| API | Use it when | Returns |
| --- | --- | --- |
| `relay.claude.spawn(...)`, `relay.codex.spawn(...)`, `relay.gemini.spawn(...)`, `relay.opencode.spawn(...)` | The provider is fixed in code | `Agent` |
| `relay.spawn(name, cli, task?, options?)` | The provider is chosen dynamically or you need advanced options | `Agent` |
| `relay.spawnAndWait(...)` / `relay.spawn_and_wait(...)` | The next step depends on the agent being ready before you continue | `Agent` |

<Note>
All of these APIs return an `Agent` handle. After spawn, the most common next steps are `waitForReady` / `wait_for_ready`, `sendMessage` / `send_message`, `onOutput` / `on_output`, and `release`.
</Note>

<Note>
If you omit `channels`, spawned agents default to `general`. Set channels explicitly when you want workflow isolation. Also note that `opencode` uses a headless runtime, while the other built-in shorthand helpers are PTY-backed.
</Note>

## Spawn a named worker

<CodeGroup>
```typescript TypeScript file="spawn-agent.ts"
import { AgentRelay, Models } from '@agent-relay/sdk';

const relay = new AgentRelay({ channels: ['dev'] });

const planner = await relay.claude.spawn({
  name: 'Planner',
  model: Models.Claude.SONNET,
  channels: ['dev'],
  task: 'Break the work into 3 implementation steps.',
});

await planner.waitForReady();

const system = relay.system();
await system.sendMessage({
  to: 'Planner',
  text: 'Start by proposing the plan, then hand implementation to Coder.',
});
```

```python Python file="spawn_agent.py"
from agent_relay import AgentRelay, Models

relay = AgentRelay(channels=["dev"])

planner = await relay.claude.spawn(
    name="Planner",
    model=Models.Claude.SONNET,
    channels=["dev"],
    task="Break the work into 3 implementation steps.",
)

await planner.wait_for_ready()

system = relay.system()
await system.send_message(
    to="Planner",
    text="Start by proposing the plan, then hand implementation to Coder.",
)
```
</CodeGroup>

## Choosing the spawn API

- Use `relay.claude.spawn(...)`, `relay.codex.spawn(...)`, and the other shorthand helpers when you already know the CLI you want.
- Use `relay.spawn(name, cli, task?, options?)` when the CLI is chosen dynamically at runtime or when you need advanced relay-managed options.
- Use `spawnAndWait` / `spawn_and_wait` when the next step depends on the agent being ready before you continue.
- Use `waitForMessage` / `wait_for_message` on `spawnAndWait` when the next step depends on the agent producing its first reply, not just reaching the ready state.

## Built-in shorthand spawners

<CodeGroup>
```typescript TypeScript file="spawners.ts"
const claudeWorker = await relay.claude.spawn({ name: 'Planner' });
const codexWorker = await relay.codex.spawn({ name: 'Coder' });
const geminiWorker = await relay.gemini.spawn({ name: 'Researcher' });
const opencodeWorker = await relay.opencode.spawn({ name: 'Reviewer' });
```

```python Python file="spawners.py"
claude_worker = await relay.claude.spawn(name="Planner")
codex_worker = await relay.codex.spawn(name="Coder")
gemini_worker = await relay.gemini.spawn(name="Researcher")
opencode_worker = await relay.opencode.spawn(name="Reviewer")
```
</CodeGroup>

These are the fastest way to spawn when the provider is fixed in code. If the provider is selected dynamically, use the generic API below.

## Generic `relay.spawn(...)`

<CodeGroup>
```typescript TypeScript file="dynamic-spawn.ts"
const cli = process.env.AGENT_CLI ?? 'claude';

const reviewer = await relay.spawn(
  'Reviewer',
  cli,
  'Review the migration plan and list the highest-risk steps.',
  {
    channels: ['review'],
    model: 'sonnet',
    cwd: '/repo',
  }
);
```

```python Python file="dynamic_spawn.py"
from agent_relay import SpawnOptions

cli = "claude"

reviewer = await relay.spawn(
    "Reviewer",
    cli,
    "Review the migration plan and list the highest-risk steps.",
    SpawnOptions(
        channels=["review"],
        model="sonnet",
        cwd="/repo",
    ),
)
```
</CodeGroup>

Use this form when:

- The CLI is chosen from config or user input
- You want one code path that can spawn different providers
- You need advanced options such as `team`, `shadowOf` / `shadow_of`, `idleThresholdSecs` / `idle_threshold_secs`, or `restartPolicy` / `restart_policy`

## `spawnAndWait` / `spawn_and_wait`

<CodeGroup>
```typescript TypeScript file="spawn-and-wait.ts"
const analyst = await relay.spawnAndWait(
  'Analyst',
  'claude',
  'Summarize the codebase structure before implementation starts.',
  {
    channels: ['planning'],
    timeoutMs: 30_000,
    waitForMessage: false,
  }
);
```

```python Python file="spawn_and_wait.py"
from agent_relay import SpawnOptions

analyst = await relay.spawn_and_wait(
    "Analyst",
    "claude",
    "Summarize the codebase structure before implementation starts.",
    SpawnOptions(channels=["planning"]),
    timeout_ms=30_000,
    wait_for_message=False,
)
```
</CodeGroup>

- `waitForMessage: false` / `wait_for_message=False` waits for broker readiness.
- `waitForMessage: true` / `wait_for_message=True` waits for the agent's first message instead.

## Common spawn options

These options are available on the shorthand helpers and on `relay.spawn(...)`:

| Option | TypeScript | Python | What it does |
| --- | --- | --- | --- |
| Name | `name` | `name` | Stable identity other agents can message |
| Model | `model` | `model` | Model string or enum for that provider |
| Task | `task` | `task` | Initial prompt for autonomous startup |
| Channels | `channels` | `channels` | Rooms the agent joins on spawn |
| CLI args | `args` | `args` | Extra CLI arguments |
| Working directory | `cwd` | `cwd` | Per-agent working directory override |
| Skip relay prompt | `skipRelayPrompt` | `skip_relay_prompt` | Skip MCP/protocol prompt injection when relay messaging is not needed |
| Lifecycle hooks | `onStart`, `onSuccess`, `onError` | `on_start`, `on_success`, `on_error` | Run code before spawn, after success, or on error |

## Advanced options on `relay.spawn(...)`

These are available on the generic API and `SpawnOptions`, not on the shorthand helpers:

| Option | TypeScript | Python | What it does |
| --- | --- | --- | --- |
| Team routing | `team` | `team` | Attach the agent to a team-aware workflow or relay-managed grouping |
| Shadow source | `shadowOf` | `shadow_of` | Spawn a shadow worker linked to another agent |
| Shadow mode | `shadowMode` | `shadow_mode` | Configure shadow behavior |
| Idle threshold | `idleThresholdSecs` | `idle_threshold_secs` | Control when idle lifecycle events fire |
| Restart policy | `restartPolicy` | `restart_policy` | Configure broker-managed restart behavior |

## What to do after spawn

- Wait for readiness with `agent.waitForReady()` / `agent.wait_for_ready()` or by name with `relay.waitForAgentReady(name)` / `relay.wait_for_agent_ready(name)`.
- Send the first message from `relay.system()` or a named human handle with `relay.human(...)`.
- Subscribe to output with `agent.onOutput(...)` / `agent.on_output(...)` if you want live PTY output.
- Use `waitForIdle()` / `wait_for_idle()` when you care about the first idle point after work begins.
- Use `release()` when the worker is no longer needed.

## Common patterns

- Omit `task` when you want to spawn a passive worker and message it later.
- Include `task` when the agent should begin autonomously as soon as it starts.
- Set `channels` explicitly on every worker in a collaboration so they all join the same room.
- Prefer `spawnAndWait` when the next step is blocked on startup; otherwise spawn first and wait later in parallel with other setup.
